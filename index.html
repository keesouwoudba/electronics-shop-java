📐 Architecture Overview

Access Modifier Guidelines

Java Access Modifiers (from most to least restrictive):

  • private: Accessible only within the same class
  • default (no modifier): Accessible within the same package (package-private)
  • protected: Accessible within the same package AND by subclasses
  • public: Accessible from anywhere

Our Security Policy

  • Fields: Always private (encapsulation)
  • Internal helper methods: private
  • Methods called by other classes: public
  • Model classes: public (need to pass between layers)
  • Repository classes: public (dependency injection)
  • Service classes: public (dependency injection)
  • Static usage: ONLY in MockDatabase (simulates shared database)

Project Philosophy

Layered Architecture with Separation of Concerns

The application is structured into 4 distinct layers:

  1. Model Layer - Pure data structures (POJOs) with no logic
  2. Repository Layer - Data access and storage management (currently in-memory arrays)
  3. Service Layer - Business logic and rules (the "brain" of the application)
  4. View Layer - User interface and input/output handling (Console-based)

Key Design Principles

  • Dependency Injection: Services receive repositories via constructor
  • No Cross-Layer Dependencies: View never talks to Repository directly
  • Scalability: Repository layer can be swapped for file/database without touching Service/View
  • Single Responsibility: Each class has one clear purpose
  • Minimal Static: Only MockDatabase uses static (simulates singleton database)
  • Maximum Encapsulation: All fields private, access via methods only

Checkout Process Algorithm:

1. Check authService.isLoggedIn()
   If false, return "NOT_LOGGED_IN"

2. Get user's cart via cartRepository.findByUserId()

3. Check cart.getItemCount() > 0
   If 0, return "EMPTY_CART"

4. VALIDATION PHASE (check all items before modifying anything)
   For each item in cart.getItems()[0...itemCount-1]:
     - Fetch fresh Product from productRepository.findById()
     - Check if product still exists (not deleted)
     - Check if product.stockQuantity >= item.quantity
     - If ANY item fails, return "STOCK_ERROR:"

5. STOCK DEDUCTION PHASE (only if all validations passed)
   For each item in cart:
     - productRepository.updateStock(item.productId, -item.quantity)

6. ORDER CREATION PHASE
   - Generate orderId via orderRepository.getNextOrderId()
   - Get current timestamp
   - Calculate totalPrice via cart.getTotalPrice()
   - Copy cart items to new OrderItem[] array
   - Create Order object with status="COMPLETED"
   - orderRepository.save(order)

7. CART CLEARING PHASE
   - cartRepository.clear(currentUser.getUserId())

8. Return "SUCCESS"

public static void main(String[] args) {
    // 1. Display initialization messages
    System.out.println("==========================================");
    System.out.println("   ELECTRONICS SHOP - INITIALIZING");
    System.out.println("==========================================");
    System.out.println();
    System.out.println("Loading data...");
    
    // 2. Initialize Repositories (triggers static blocks)
    UserRepository userRepo = new UserRepository();
    ProductRepository productRepo = new ProductRepository();
    OrderRepository orderRepo = new OrderRepository();
    CartRepository cartRepo = new CartRepository();
    
    System.out.println("✓ Data loaded");
    System.out.println("Initializing services...");
    
    // 3. Initialize Services (dependency injection)
    AuthService authService = new AuthService(userRepo);
    ShopService shopService = new ShopService(productRepo, orderRepo, cartRepo, authService);
    AdminService adminService = new AdminService(productRepo, userRepo, orderRepo, authService);
    
    System.out.println("✓ Services initialized");
    System.out.println("Starting application...");
    System.out.println();
    
    // 4. Wait 1 second
    try { Thread.sleep(1000); } catch (InterruptedException e) {}
    
    // 5. Initialize View
    ConsoleUI ui = new ConsoleUI(authService, shopService, adminService);
    
    // 6. Start application loop (runs until System.exit(0))
    ui.start();
    
    // 7. This line only reached after System.exit()
    System.out.println("Application terminated.");
}

📊 Summary

Architecture Recap

4-Layer Architecture with Maximum Encapsulation:

1. Model Layer (6 classes)

  • All fields: private
  • Getters/Setters: public
  • Classes: public (data transfer)
  • Exception: MockDatabase uses public static (simulates DB)

2. Repository Layer (4 classes)

  • Classes: public (DI)
  • All methods: public (API for Service)
  • No instance fields
  • Static init blocks for seeding

3. Service Layer (3 classes)

  • Classes: public (DI)
  • Fields: private (encapsulation)
  • Business methods: public
  • Helpers: private
  • No static

4. View Layer (1 class)

  • Class: public
  • Fields: private
  • start(): public (Main entry)
  • All UI methods: private
  • One static constant

Key Features

User Features:

  • Browse products (pagination, 10/page, Z/M navigation)
  • View product details
  • Add to basket (with quantity)
  • Remove from basket (numbered selection)
  • Register (password validation: 6+ chars, letters + digits)
  • Login/Logout
  • Checkout (stock validation)

Admin Features:

  • Add/update/delete products
  • Manage stock (add/remove partial quantities)
  • Change prices/names/discounts
  • Add/edit/delete users
  • View statistics (product/user/order counts, revenue)
  • Role-based access control

Technical Features:

  • Session management (persistent login)
  • Application loop (doesn't exit on logout)
  • Cart persistence (per user)
  • Stock validation (prevents overselling)
  • Transaction integrity (validates all before any deduction)
  • Maximum encapsulation
  • Minimal static usage

Static Usage Justification

Only MockDatabase uses static:

  • Simulates singleton database instance
  • Provides shared storage across application
  • Will be replaced by actual database connection (still singleton)

Everything else is non-static:

  • Services are instantiated (testable, flexible)
  • Repositories are instantiated (swappable data sources)
  • View is instantiated (multiple UI instances possible)

Security & Encapsulation Benefits

  1. Data Hiding: Private fields prevent unauthorized access
  2. Controlled Access: Public methods are gatekeepers with validation
  3. Minimal Static: Prevents global state issues
  4. Layer Isolation: Private methods prevent layer-hopping
  5. Dependency Injection: Testable, modular, flexible code
